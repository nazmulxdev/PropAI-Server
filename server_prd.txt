SERVER-SIDE PRODUCT REQUIREMENTS DOCUMENT
Version 1.0 | Backend Architecture, API Design, Database, and AI Integration


1. Server Overview
The PropAI backend is a RESTful API server built with Node.js, Express, and TypeScript. It powers all data operations, authentication, real-time communication, AI feature orchestration, and background job processing for the PropAI real estate platform. The server is designed for scalability, reliability, and production-readiness from the ground up.

Core Responsibilities:
Authentication & authorization (JWT + role-based access control).
CRUD operations for all entities: users, properties, inquiries, messages, blogs, reviews, notifications.
AI API orchestration: Claude/Gemini calls for all 4 AI features.
RAG (Retrieval-Augmented Generation) pipeline for the AI chatbot.
Real-time communication via Socket.io.
Background job processing via BullMQ.
Rate limiting, logging, caching, and error tracking.

7.2 AI Smart Recommendations — /api/ai/recommendations
7.2.1 POST /api/ai/recommendations
Auth: Buyer only.
How the endpoint works:
Step 1: Fetch user's browsing history (last 15 viewed property IDs) from BrowsingHistory table.
Step 2: Fetch user's saved property IDs.
Step 3: Fetch user's profile preferences (preferredCities, preferredTypes, priceRange).
Step 4: Fetch 60 most recent ACTIVE property IDs from database (candidates).
Step 5: Build prompt for Claude with user context + candidate property summaries.
Step 6: Claude returns 8 recommended property IDs with reasoning.
Step 7: Fetch full property data for those IDs. Return to client.

Prompt Construction:
User Profile: Prefers {types} in {cities}, budget {min}-{max} BDT.
Recently viewed: {title, type, city, price for each viewed property}.
Saved properties: {title, type, city for each saved}.
Available properties to recommend from: {id, title, type, city, price, bedrooms, amenities}.
Return ONLY JSON: { recommendations: [ { id: string, reason: string } ] } — 8 items.

Request Body: {} (server fetches everything from DB using JWT user ID)
Response:
{
  recommendations: [
    {
      property: PropertyObject,  // full property with seller info
      reason: string             // AI-generated reason: 'Matches your preference for...'
    }
  ]
}
Caching: Cache result per user in Redis for 30 minutes. Key: ai:recommendations:{userId}. Invalidate on new browse or save event.

7.4 RAG Chatbot — /api/ai/chat
7.4.1 POST /api/ai/chat/message
Auth: Public (but user context added if logged in for personalization).
Request Body:
{
  message: string,              // user's question
  conversationHistory: [        // last 6 messages for context
    { role: 'user' | 'assistant', content: string }
  ]
}
RAG Pipeline Steps on the server:
Step 1 — Intent Extraction: Parse user message to extract: location mentions, property type, price range, number of bedrooms, listing type (sale/rent).
Step 2 — Database Retrieval: Query PostgreSQL with extracted filters. Retrieve up to 8 matching ACTIVE properties. Fields: id, title, city, area, type, price, bedrooms, bathrooms, areaSqft, amenities, listingType.
Step 3 — Context Building: Format retrieved properties into a structured string for the AI context.
Step 4 — AI Call: Send to Claude with system prompt: 'You are PropAI's helpful real estate assistant. Answer ONLY based on the provided property data. If no matching properties exist in the data, say so clearly. Do not invent properties. Be friendly and specific.'
Step 5 — Response: Return AI's natural language response to client.

Response:
{
  reply: string,       // AI's natural language response
  properties: [...]    // array of property IDs mentioned — frontend fetches full data
}
Rate Limiting: 20 messages per 10 minutes per IP/user to prevent AI API cost abuse.

8.4 Global Error Handler
Single error-handling middleware at the end of the Express middleware chain:
Catches all errors passed via next(error).
Distinguishes between operational errors (known, safe to expose) and programming errors (unknown).
For operational errors (ZodError, Prisma errors, custom AppError): returns structured JSON with status code and message.
For unknown errors: returns 500 with generic message. Full error sent to Sentry.
Never exposes stack traces in production.
Response format for all errors:
{
  success: false,
  error: {
    code: string,     // e.g. 'VALIDATION_ERROR', 'NOT_FOUND', 'UNAUTHORIZED'
    message: string,  // human-readable message
    fields?: { [field: string]: string }  // only for validation errors
  }
}

10.1 Cache Implementation Pattern
// Generic cache-aside pattern
async function getCachedOrFetch<T>(key: string, ttl: number, fetcher: () => Promise<T>): Promise<T> {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);
  const data = await fetcher();
  await redis.setex(key, ttl, JSON.stringify(data));
  return data;
}

12.3 Job Error Handling
All jobs have a retry policy: 3 attempts with exponential backoff (1s, 5s, 30s).
After 3 failures: job moved to dead letter queue, error logged to Winston + Sentry.
Email job failures are silent to user — never show email failure in API response.


13. Error Tracking (Sentry)
Sentry initialized at app startup with SENTRY_DSN from environment.
Sentry Express integration captures: all unhandled exceptions, Express route errors, slow transactions (> 2s).
User context attached when available: { id, email, role } from req.user.
Custom tags added: { environment, feature } — e.g., feature: 'ai-recommendations'.
Sensitive data scrubbing: passwords, tokens, API keys never sent to Sentry. Configured via denyUrls and before-send hook.
Performance monitoring: tracesSampleRate: 0.1 (10% of transactions) in production to avoid cost.


14. Authentication & Security System
14.1 JWT Strategy
Token
Expiry
Storage
Rotation
Access Token
15 minutes
Client memory / Authorization header
Refreshed automatically when expired using refresh token

7 days
httpOnly, Secure, SameSite=Strict cookie

14.2 Password Security
Passwords hashed with bcrypt, salt rounds: 12.
Password reset tokens: crypto.randomBytes(32) hashed with SHA256 before DB storage.
Reset token expiry: 1 hour.
Minimum password requirements enforced: 8 chars, 1 uppercase, 1 lowercase, 1 number.

14.3 Security Headers (Helmet)
Content-Security-Policy: strict, allowing only own origin and Cloudinary for images.
X-Frame-Options: DENY (prevents clickjacking).
X-Content-Type-Options: nosniff (prevents MIME type sniffing).
Referrer-Policy: strict-origin-when-cross-origin.
HSTS: max-age 1 year, includeSubDomains (production only).

14.4 Input Validation
All request bodies validated with Zod schemas before reaching controller.
Validation middleware wraps all route handlers.
Unknown fields are stripped (strict Zod schemas). No extra data passes through.
SQL injection: impossible with Prisma parameterized queries.
File upload validation: MIME type + extension must match. Max file size: 5MB per image.
